# Documentation

For whoever writes docs, dictator or worker.

## Build

- Open every doc with a top level description of its purpose
- Write the current truth, no changelogs or stale content
- Communicate the high level objective, implementation detail can be path-found by frontier models
- Prefer dotpoints over verbose paragraphs
- No em-dashes or semicolons
- No hardcoded file references in project docs

## Subtract

For each section and line:

- Identify what a reader can only learn from it
- Remove it if the doc still serves its purpose without it

## Verify

The doc is complete when:

- Every line earns its keep
- Nothing is stale or contradicts the current state of the project
- Further removal would lose information a reader needs
